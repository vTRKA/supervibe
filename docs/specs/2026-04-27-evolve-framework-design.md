# Supervibe Framework — Design Spec

Claude Code plugin: specialist agents, code graph, project memory, confidence gates, and stack-aware scaffolding.

This is the **META-spec**. Each implementation phase (see Section 8) gets its own sub-spec when reached.

## Locked decisions

| # | Decision | Reason |
|---|----------|--------|
| 1 | **Distribution = Claude Code plugin** in `D:\ggsel projects\supervibe\` | "use across many projects as superpowers" |
| 2 | **Full process toolkit** (own brainstorming/plan/exec/TDD/debug/review) | Maximum-effort own solution |
| 3 | **Content language = English** | Tech precision, no Russian/English mix in artifacts |
| 4 | **Confidence policy = HARD BLOCK + escape hatch** | Discipline by default, `/supervibe override "<reason>"` for exceptions, full audit trail |
| 5 | **Build order = C → A → B → E → orchestrator** | Quality contract first, then catalog, then scaffold, then evolution loop, then orchestration |
| 6 | **Architecture = Hybrid plugin** (flat skills, namespaced agents, stack-packs) | Approach 3 from brainstorm — best balance of discoverability and scale |


### Two activation modes

1. **User-invoked** — slash commands in `commands/`
2. **Claude-invoked (proactive)** — via skill `description` field; main agent reads description and self-invokes when trigger phrase matches

### `registry.yaml` contract

Auto-generated by `scripts/build-registry.mjs` on every change to `agents/`, `rules/`, `skills/`, `stack-packs/`. Format:

```yaml
version: 1.0.0
generated-at: <ISO timestamp>
agents:
  supervibe:<namespace>:<name>:
    file: <path>
    capabilities: [...]
    stacks: [...] | [any]
    requires: [...]    # optional
skills: ...
rules: ...
stack-packs:
  <pack-id>:
    manifest: <path>
    stacks: [...]
    architectures: [...]
    optional: [...]
```

### Lifecycle: empty repo → working scaffold

```
1. User opens project, says "let's start" / "help with current code"
2. Main agent reads supervibe:stack-discovery description, self-invokes it
3. supervibe:stack-discovery:
   ├─ Scan manifests (package.json, composer.json, Cargo.toml, etc.)
   ├─ If stack detected → confirm with user
   ├─ If empty → invoke supervibe:requirements-intake (questionnaires)
   └─ Output: stack-fingerprint
4. supervibe:requirements-intake (if needed):
   ├─ Stack-aware questions one-at-a-time
   ├─ Architecture, infra, design, testing, deployment
   └─ confidence-scoring(requirements-spec) ≥9 — else more questions
5. supervibe:genesis:
   ├─ Pick stack-pack from registry (or compose from atomic packs)
   ├─ Copy pack/claude/* → project/.claude/
   ├─ Copy pack/configs/* → project root
   ├─ Copy pack/husky/* → project/.husky/
   ├─ Generate CLAUDE.md from template + discovery results
   ├─ Generate settings.json (allow/ask/deny)
   └─ confidence-scoring(scaffold-bundle) ≥9
6. supervibe:evaluate (after first task)
   └─ Track effectiveness, suggest strengthen/adapt as needed
```

## Section 2 — Confidence Engine

### Five artifact types (each with own rubric)

| Artifact | Scored when | Rubric |
|----------|-------------|--------|
| `requirements-spec` | After requirements-intake/brainstorming | `confidence-rubrics/requirements.yaml` |
| `implementation-plan` | After writing-plans, before exec | `confidence-rubrics/plan.yaml` |
| `agent-output` | When agent claims "done" | `confidence-rubrics/agent-delivery.yaml` |
| `scaffold-bundle` | After genesis composes scaffold | `confidence-rubrics/scaffold.yaml` |
| `framework-self` | On `/supervibe audit` of plugin itself | `confidence-rubrics/framework.yaml` |

Plus `prototype.yaml`, `research-output.yaml`, `rule-quality.yaml`, `skill-quality.yaml`, `agent-quality.yaml` for sub-artifacts.

### Rubric format (YAML)

```yaml
artifact: <type>
max-score: 10
dimensions:
  - id: <slug>
    weight: <int, sum across dimensions = 10>
    question: "<assessment question>"
    evidence-required: "<what proves the dimension passes>"
gates:
  block-below: 9     # default for v1.0
  warn-below: 10
```

5 dimensions × 2 weight = 10. Default gate ≥9 across all artifact types (per "10/10 = done" requirement).

### `supervibe:confidence-scoring` skill

```
INPUT:  artifact-type, artifact-content, evidence
PROCESS:
  1. Load rubric
  2. For each dimension, evaluate against evidence (no evidence = 0)
  3. Sum weighted score
  4. Decide: BLOCK (<block-below), WARN (block≤s<warn), PASS (≥warn)
OUTPUT: { score, max, status, gaps: [{dim, missing}] }
```

### HARD BLOCK + escape hatch

- Agent CANNOT claim "done" without scoring ≥9. Enforced by `rules/confidence-discipline.md` (mandatory in every project).
- `/supervibe override "<reason>"` allows ship at <9 with append-only entry to `.claude/confidence-log.jsonl`:
  ```json
  {"timestamp":"<ISO>","artifact":"<type>","score":7,"max":10,"override":true,"reason":"<text>","gaps":[...],"agent":"<id>","user-confirmed":true}
  ```
- `supervibe:audit` reads log: override-rate >5% across last 100 → flag systemic issue.

### Self-scoring of framework

`supervibe:audit` periodically scores plugin itself by `framework.yaml`:
- `stack-coverage`: how many top-20 stacks have full pack
- `agent-quality`: how many agents <250 lines or missing Persona
- `skill-quality`: how many skills <80 lines or missing decision tree
- `rule-freshness`: how many rules with last-verified > 90 days
- `discipline`: override-rate < 5% across last 100 artifacts

`framework-self` < 9 → auto-suggests strengthen/adapt.

## Section 3 — Knowledge Base

### Quality bar (non-negotiable)

- **Agent**: ≥250 lines (8-15 KB), 15-year Persona, concrete anti-patterns, concrete verification commands, concrete scope
- **Skill**: ≥80 lines, mandatory Step 0 (read source of truth), decision tree, output contract, guard rails
- **Rule**: ≥200 lines with good/bad examples, "why" reasoning, related-rules links

Below bar → not shipped. `supervibe:audit` flags WEAK, `supervibe:strengthen` mandatory.

### Universal frontmatter

```yaml
---
name: <name>
namespace: <_core | _design | _product | _ops | _meta | stacks/{stack}>
description: "<USE WHEN <trigger>... TO <purpose>... GATES <scoring>>"
persona-years: 15
capabilities: [...]
stacks: [...] | [any]
requires-stacks: [...]    # optional
optional-stacks: [...]    # optional
tools: [Read, Grep, Glob, ...]
skills: [supervibe:...]
verification: [...]
anti-patterns: [...]
version: 1.0
last-verified: <date>
verified-against: <commit-hash>
effectiveness:            # filled by supervibe:evaluate
  last-task: null
  outcome: null
  iterations: 0
---
```

### Agents — namespace map

(Full list in Section 3 of brainstorm transcript, summarized here.)

- `_core/` — code-reviewer, root-cause-debugger, repo-researcher, security-auditor, refactoring-specialist, architect-reviewer, quality-gate-reviewer
- `_design/` — creative-director, ux-ui-designer, ui-polish-reviewer, accessibility-reviewer, copywriter, prototype-builder
- `_product/` — product-manager, systems-analyst, qa-test-engineer, analytics-implementation, seo-specialist, email-lifecycle
- `_ops/` — devops-sre, performance-reviewer, dependency-reviewer, db-reviewer, api-contract-reviewer, infrastructure-architect, **5 research-agents** (best-practices, dependency, security, infra-pattern, competitive-design)
- `_meta/` — rules-curator, supervibe-orchestrator
- `stacks/{stack}/` — per stack: architect (read-only design), developer (write), specialists as needed

### Skills — categories

**Process** (own, replace superpowers): brainstorming, writing-plans, executing-plans, tdd, systematic-debugging, code-review, verification, requirements-intake, requesting-code-review, receiving-code-review, dispatching-parallel-agents, subagent-driven-development, using-git-worktrees, finishing-a-development-branch, pre-pr-check

**Capability**: confidence-scoring, stack-discovery, adr, prd, prototype, landing-page, new-feature, incident-response, experiment, rule-audit, sync-rules, seo-audit

**Evolution**: genesis, audit, strengthen, adapt, evaluate

### Rules — initial set (port from product-framework + new)

Ported & generalized: best-practices-2026, fsd, modular-backend, routing, i18n, observability, privacy-pii, rule-maintenance.

New for supervibe: git-discipline, commit-discipline, no-dead-code, confidence-discipline, anti-hallucination, infrastructure-patterns, prototype-to-production, pre-commit-discipline.

### Skill vs Agent distinction

- **Skill** = methodology (how to do). Read-only by default. Attachable to multiple agents.
- **Agent** = executor (who does). Has tools, scope, attached skills.

Same skill can be attached to multiple agents (e.g., `supervibe:code-review` attached to code-reviewer agent AND to laravel-developer for self-review).

## Section 4 — Discovery & Scaffolding

### Questionnaire format

YAML banks in `questionnaires/`. Each question: `id`, `when` (condition), `type` (single/multi-choice), `text`, `options[].id`, `options[].label`, optional `recommends` (agents to attach), `attaches-rule`.

Initial banks: 01-stack-foundation, 02-architecture, 03-infra, 04-design, 05-testing, 06-deployment.

### Discovery flow

1. Scan project files (Glob manifests)
2. Extract detected stack from manifests
3. Load questionnaires ordered by priority
4. Ask each question with `when` matching, one at a time
5. Skip auto-detected with confirmation
6. Build stack-fingerprint
7. confidence-scoring(requirements-spec) ≥9
8. Handoff to genesis with fingerprint

### Stack-pack manifest

```yaml
id: <pack-id>
name: <human readable>
version: 1.0.0
matches:
  required: { backend, frontend, db }
  optional: { infra, architecture }
agents-attach: [supervibe:...]      # what gets copied to target/.claude/agents/
rules-attach: [...]
scaffold:
  claude:
    settings.json: { template: ... }
    CLAUDE.md: { template: ..., placeholders: [...] }
  root-files: [...]
  husky: { pre-commit, commit-msg, pre-push }
  directories: [...]
post-genesis-actions: [...]
```

### Pack composition (no exact match)

1. Find base pack with max overlap
2. For each missing slot, pull atomic-packs (`stack-packs/_atomic/{redis,queue,db-replicas,...}`)
3. Merge: union(agents-attach), union(rules-attach), merge(scaffold) with conflict resolution
4. confidence-scoring(scaffold-bundle); <9 → user confirm
5. Write

### Pre-commit pipeline (default in every scaffold)

| Hook | Action | Owner |
|------|--------|-------|
| `pre-commit` | lint-staged → format + lint + type-check incremental | husky + lint-staged |
| `commit-msg` | commitlint (Conventional Commits) | husky + commitlint |
| `pre-push` | full typecheck + fast tests + dep audit | husky |
| CI | full lint + tests + e2e + security + build | GitHub Actions |

Hooks composed by generator from `templates/husky/{stack}` per pack manifest.

### Prototype workflow

```
supervibe:prototype (triggered by "design X")
1. creative-director → visual direction
2. ux-ui-designer → screen spec (component inventory, states matrix)
3. prototype-builder → 1:1 HTML/CSS in prototypes/{feature}/
4. ui-polish-reviewer → 8-dim review
5. confidence-scoring(prototype) ≥9
6. USER APPROVAL (open index.html)
7. Handoff to frontend developer for 1:1 transfer
8. Post-transfer: ui-polish-reviewer compares prototype vs implementation; fail if drift >5%
```

## Section 5 — Process Skills mechanics

### Skill contract

```yaml
---
name: <name>
namespace: process
description: "<trigger phrase + purpose + gate>"
allowed-tools: [...]
phase: brainstorm | plan | exec | review
prerequisites: [...]
emits-artifact: <type>
confidence-rubric: confidence-rubrics/<name>.yaml
gate-on-exit: true
version: 1.0
---
```

`gate-on-exit: true` — skill cannot complete without invoking confidence-scoring. Enforced by `confidence-discipline.md`.

### Key process skills (mechanics summary)

- **`supervibe:brainstorming`** — like superpowers but: SCOPE-check for multi-subsystem, stack-aware questions, confidence-gate ≥9 on requirements-spec before transition. NO visual companion in v1.0.
- **`supervibe:writing-plans`** — phased plan with per-phase verification commands and confidence-gate. ≥9 on plan before exec.
- **`supervibe:executing-plans`** — phase-by-phase with mandatory verification; ≥9 per phase before next.
- **`supervibe:tdd`** — red-green-refactor; integration > mocks where rule says so; confidence-gate before commit.
- **`supervibe:systematic-debugging`** — symptom → max-3 hypothesis → evidence → isolation → minimal fix → verify; gate at minimal-fix.
- **`supervibe:verification`** — RULE: no claim without command output. Always invoked before claim.
- **`supervibe:requirements-intake`** — entry gate. Decides complexity:
  - ≥7 → brainstorming
  - 3-6 → writing-plans direct
  - ≤2 → straight to executing (after triviality check)

### Confidence-gate map

```
[user request]
  → supervibe:requirements-intake → GATE: requirements-spec ≥9
  → [if complexity≥7] supervibe:brainstorming → GATE: requirements-spec ≥9 (deeper)
  → supervibe:writing-plans → GATE: implementation-plan ≥9
  → supervibe:executing-plans
      ├─ Phase 1 → GATE: agent-output ≥9
      ├─ Phase 2 → GATE: agent-output ≥9
      └─ Phase N → GATE: agent-output ≥9
  → supervibe:requesting-code-review
  → code-reviewer agent → GATE: review-report ≥9
  → [if any GATE fails AND no override] LOOP
  → [if override] log to .claude/confidence-log.jsonl
  → done; supervibe:evaluate runs async
```

## Section 6 — Agent evolution loop

### Two-level evolution

| Level | What | Where | Trigger |
|-------|------|-------|---------|
| Project | `.claude/agents/`, `.claude/rules/`, `.claude/skills/`, `CLAUDE.md` | target project | supervibe-orchestrator + auto-triggers |
| Plugin | `evolve/` itself | this repo | `/supervibe audit` run on plugin root |

Same skill set, parameterized by `target-root`.

### Five evolution skills

- **`supervibe:genesis`** — bootstrap from stack-fingerprint + composed pack. Never overwrites. ≥9 scaffold gate.
- **`supervibe:audit`** — health check: stale references (grep paths/funcs/cmds), coverage gaps (uncovered modules), weak (size/persona/decision-tree), version staleness (>30d), override-rate (>5%), effectiveness signals.
- **`supervibe:strengthen`** — deepen weak: add Persona, real paths, anti-patterns from rules+memory, verification commands, decision trees, examples. Never deletes. Bumps version.
- **`supervibe:adapt`** — sync to stack changes: stale refs, new modules → assign agent, new deps → minor update or suggest genesis.
- **`supervibe:evaluate`** — write effectiveness frontmatter (outcome, iterations, blockers, confidence-score, user-corrections). Pattern detection auto-suggests audit/strengthen.

### `rules-curator` agent

Special agent (not skill) that maintains `.claude/rules/*` of target project.

Triggered by: user "add rule X", audit-detected contradiction, repeated rule violation in effectiveness, new rule file.

Procedure: read all rules + MEMORY + new rule → format under template → cross-link → update CLAUDE.md if mandatory → update settings.json deny if introduces ban → confidence-gate ≥9 → diff → approval → write → trigger `supervibe:sync-rules`.

### `effectiveness.jsonl` (append-only)

```json
{"ts":"<ISO>","agent":"<id>","task":"<text>","outcome":"success|partial|failed","iterations":<int>,"blockers":[...],"confidence":<int>,"user-corrections":<int>}
```

Sorted by rolling-average outcome → strengthen worst first.

### Auto-triggers

| Trigger | Phase | Detection |
|---------|-------|-----------|
| `.claude/agents/` empty / no routing | genesis | session start |
| New dir in src/modules | adapt | post-Write hook |
| New major dependency | adapt | post-edit on manifests |
| Agent failed stale-context | audit | effectiveness write |
| >10 files changed in session | audit | session activity |
| Oldest `last-verified` >30d | audit + strengthen | session start |
| New rule file | strengthen + sync-rules | post-Write |
| override-rate >5% | audit + strengthen | session start |
| User corrected agent 2+ times | evaluate + strengthen | conversation analysis |

Triggers do NOT auto-execute state-changing actions. They emit system-reminders → main agent proposes to user → user confirms.

## Section 7 — Orchestration & Research

### Three auto-invocation mechanisms

1. **Skill descriptions** — main agent reads `description`, self-invokes when trigger matches.
2. **settings.json hooks** — Claude Code harness runs shell commands on events (SessionStart, PostToolUse, Stop).
3. **`supervibe-orchestrator` agent** — meta-decision-maker reading weighted context.

### Description template (mandatory format)

```
USE [WHEN <trigger phrase from user>] [TO <verb-led purpose>] [GATES <scoring/check>]
```

`supervibe:audit` rubric `skill-quality` has dimension `trigger-clarity`; <2 = block.

### Hooks (in scaffolded `settings.json`)

```jsonc
{
  "hooks": {
    "SessionStart": [{ "matcher": "*", "hooks": [{ "type": "command", "command": "node $CLAUDE_PLUGIN_ROOT/scripts/session-start-check.mjs" }]}],
    "PostToolUse":  [{ "matcher": "Write|Edit", "hooks": [{ "type": "command", "command": "node $CLAUDE_PLUGIN_ROOT/scripts/post-edit-stack-watch.mjs" }]}],
    "Stop":         [{ "matcher": "*", "hooks": [{ "type": "command", "command": "node $CLAUDE_PLUGIN_ROOT/scripts/effectiveness-tracker.mjs" }]}]
  }
}
```

- `session-start-check.mjs` — checks last-verified, override-rate; emits system-reminders
- `post-edit-stack-watch.mjs` — watches manifests + rule changes
- `effectiveness-tracker.mjs` — writes `effectiveness.jsonl` from transcript analysis

### `supervibe-orchestrator` agent

Inputs: system-reminders, effectiveness, confidence log, user message text, stack-fingerprint.

Decides which evolve phase to run (or none). Emits proposal to user; never auto-executes state changes.

### Research agents (5 in `_ops/`)

| Agent | Researches | Sources | Triggered |
|-------|-----------|---------|-----------|
| `best-practices-researcher` | current patterns/idioms per stack | context7, official docs, Awesome lists, RFCs | audit framework, new stack added, rule stale |
| `dependency-researcher` | latest stable, deprecation, migration guides | npm/composer/cargo, GH releases | adapt after major dep change |
| `security-researcher` | CVE, known vulns | npm/composer/cargo audit, GH Security Advisories | pre-pr-check |
| `infra-pattern-researcher` | current Sentinel/Cluster/replication practices | vendor docs, neutral pattern docs | scaffold with infra, adapt infra |
| `competitive-design-researcher` | market visual/UX patterns | screenshots, public DS, Awesome design | new product creative |

All cache to `.claude/research-cache/{topic}-{date}.md` (TTL 30d). All cite sources. All gated by `research-output.yaml` ≥9.

### Boundary: proactive notify vs autonomous action

| Auto (no confirm) | Requires confirm |
|-------------------|------------------|
| Run hooks, scoring, evaluate-tracking | Run genesis (writes files) |
| Append to logs (effectiveness, confidence) | Run strengthen (modifies agents) |
| Append to research-cache (TTL) | Run adapt (modifies agents/paths) |
| Analyze diff, compute metrics | Create commits, branches, push |
| Emit system-reminders | Install deps, DB migrations |
| Read all project files | Any source/config/.claude/ change |

**Rule: proactively OBSERVE, never proactively MUTATE.**

```

```

Thin wrappers; logic in skills.

### Implementation cycle per phase

This META-spec gives decomposition. Each phase, when reached, runs its own cycle:
1. Phase-specific brainstorm (only if complexity ≥7)
2. Phase-specific spec
3. Phase-specific plan
4. Execution with per-phase confidence-gate
5. Evaluation
