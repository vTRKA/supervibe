# i18n

## Why this rule exists

Adding i18n after the fact requires touching every UI string. Done from start, costs almost nothing. Done late, costs weeks.

## When this rule applies

- Apps with multi-language audience
- Apps with multi-region audience (date/number/currency formatting)
- Apps subject to localization regulations (e.g., Quebec, EU multi-language requirements)

## What to do

### Key naming

Hierarchical, dot-separated:
```
auth.login.title
auth.login.error.invalid_credentials
billing.invoice.status.paid
```

### Plurals

Use ICU MessageFormat (or framework equivalent):
```
"items_count": "{count, plural, =0{no items} one{1 item} other{# items}}"
```

### Interpolation

Named placeholders, not positional:
```
"welcome": "Hello, {name}!"   // good
"welcome": "Hello, %s!"       // bad (translator confusion)
```

### Date / number / currency

Use Intl API (or framework wrapper):
```js
new Intl.DateTimeFormat('en-US').format(date)
new Intl.NumberFormat('de-DE', { style: 'currency', currency: 'EUR' }).format(amount)
```

### RTL support

- `dir="ltr"` / `dir="rtl"` on `<html>`
- CSS logical properties: `padding-inline-start` instead of `padding-left`
- Mirror icons (chevrons, arrows)
- Test in RTL locale

### Translation workflow

- Source in default locale
- Translation memory (xliff / po / json per locale)
- TMS integration (Crowdin / Lokalise / Phrase) for non-trivial projects
- Pseudo-localization in dev to catch hardcoded strings

## Examples

### Bad

```jsx
<button>Click here</button>   // hardcoded
{users.length} user(s)         // wrong plural handling
```

### Good

```jsx
<button>{t('auth.login.cta')}</button>
{t('users_count', { count: users.length })}
```

## Enforcement

- Linter: `eslint-plugin-i18n` or framework equivalent
- Pseudo-localization in dev catches hardcoded strings
- Code review checks new strings go through `t()`
